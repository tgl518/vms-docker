# AI 使用与提示词规范（仅针对本项目）

目的：规定与限制 AI 助手对仓库的建议与输出格式，避免 AI 擅自修改关键配置或工具文件。

## 核心原则（简明）
- AI 只能提出“建议/补丁/差异”（patch），不能直接提交、合并或替换仓库文件。
- 所有对关键文件的修改必须由人工明确批准后才能执行。
- 若请求涉及受保护文件，AI 必须先拒绝直接修改并提供替代方案与必要的审核说明。

## 受保护的文件与目录（AI 不得直接修改）
示例（非穷尽，遇到新敏感文件应先询问人工）：
- 构建/配置：vite.config.js, tsconfig.json, package.json, package-lock.json, pnpm-lock.yaml, yarn.lock
- 环境/密钥：.env, .env.*, secrets/**（任何包含凭证的文件）
- CI/部署：.github/workflows/*, .gitlab-ci.yml, Dockerfile, docker-compose.yml
- 工具/脚本：scripts/**, build/**, ci/**
- 其他：README.md 的元信息改动需谨慎，任何涉及发布/版本的变更需人工批准

## 对 AI 的输出格式（强制）
1. 步骤说明：先给 1–3 步的简短执行/建议步骤。
2. 按文件分组列出修改建议，每个被修改的文件只使用一个代码块。
3. 每个代码块的第一行必须是注释并包含完整文件路径，例如：
   - Markdown 文件：<!-- filepath: /path/to/file.md -->
   - JS/TS：// filepath: /path/to/file.ts
4. 代码块中只列出变更内容；用 `// ...existing code...` 或 `/* ...existing code... */` 来表示未改变的区域。
5. 对每个文件都要说明为什么需要修改、如何验证（包含需要新增或更新的测试用例）。
6. 禁止 AI 自动执行任何 git 操作、CI 操作或远程部署；所有操作都应由人工完成。

## 模板提示词（示例）
在向 AI 提交修改请求时，请使用下列结构化模板（中文）：
任务：简要描述目标与背景  
现有文件（或路径）：列出涉及文件路径  
预期修改：要实现的效果要点  
约束与要求：  
- 严格遵守输出格式（步骤 + 按文件分组单一代码块，首行注释包含文件路径）  
- 禁止直接修改受保护文件（若需修改，请先获得人工批准并标注为“需人工批准”）  
- 提供验证步骤与新增测试示例

示例调用：
任务：修复播放器在 iOS 上的触摸兼容问题并添加单元测试  
文件：/src/player/*.ts, /test/player.test.ts  
预期修改：修复处理触摸事件的函数并增加覆盖测试  
约束：请仅输出按照本规约的建议，不要提交或修改受保护文件；如果建议触及 vite.config.js，请标注“需人工批准”。

## 当请求触及受保护文件时（必遵流程）
- AI 应拒绝直接修改，并回复：
  - 说明为何不应直接修改（影响构建/安全/发布等）
  - 提供两个安全替代方案（例如封装变更、环境变量、或提议一个 PR 草案并标注“需人工批准”）
  - 提供明确的审查与验证 checklist（测试、CI、回滚步骤）
- 若获得人工批准，AI 的输出仍需以差异/补丁形式给出，并在变更说明中标注“已获人工批准：<审批人>”。

## 验证与合并要求
- 所有建议变更需：本地通过测试、通过 linter、CI 绿灯、并至少一名同事审阅后合并。
- 对影响范围大的变更（构建/依赖/发布）需至少两人审阅并在 PR 中详细说明风险与回滚计划。

## 拒绝策略
遇到任何涉及敏感操作（泄露密钥、绕过安全控制、未经授权的部署）或超出权限的请求，AI 应直接回复：
抱歉，我无法协助执行该操作。

---

此规约旨在把 AI 限制为“建议工具”，确保存储库安全与人类最终判断权。任何对本规约的修改需通过正常 PR 流程并获得团队同意。
